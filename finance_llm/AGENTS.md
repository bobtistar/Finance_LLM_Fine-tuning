# AGENTS.md

이 파일은 Codex가 이 프로젝트에서 코드를 수정, 생성, 검토할 때 따라야 할 작업 지침이다.

## 기본 원칙

- 성급하게 구현하지 말고, 먼저 요구사항과 주변 코드를 확인한다.
- 불확실한 요구사항, 모순된 조건, 누락된 정보가 있으면 가정을 명시한다.
- 단순하고 직접적인 해결책을 우선한다.
- 요청 범위를 벗어난 리팩터링, 포맷팅, 구조 변경을 하지 않는다.
- 변경은 작고 검증 가능해야 한다.
- 완료했다고 말하기 전에 테스트, 타입체크, 린트, 빌드 중 가능한 검증을 수행한다.

## 작업 시작 전

- 관련 파일, 기존 패턴, 테스트, 설정 파일을 먼저 읽는다.
- 기존 아키텍처와 네이밍을 따른다.
- 프로젝트에 이미 있는 유틸리티, 컴포넌트, 타입, 테스트 헬퍼를 우선 재사용한다.
- 새 의존성은 정말 필요할 때만 추가하고, 이유를 설명한다.
- 요구사항이 여러 방식으로 해석될 수 있으면 구현 전에 해석을 명시한다.

## 구현 방식

- 최소 변경으로 문제를 해결한다.
- 단일 목적의 변경만 수행한다.
- 단순한 코드가 가능하면 추상화, 팩토리, 제네릭, 설정 레이어를 추가하지 않는다.
- 한 번만 쓰이는 로직을 과도하게 일반화하지 않는다.
- 기존 코드 스타일이 완벽하지 않아도, 해당 파일의 스타일과 일관성을 우선한다.
- 관련 없는 import 정리, 파일 재정렬, 전역 포맷팅을 하지 않는다.
- 생성 파일, lockfile, 마이그레이션 파일은 명시적으로 필요할 때만 수정한다.

## 금지 사항

- 요청받지 않은 기능을 추가하지 않는다.
- 관련 없는 버그를 발견해도 임의로 고치지 말고, 별도로 언급만 한다.
- 대규모 리팩터링을 작은 버그 수정에 섞지 않는다.
- 테스트를 통과시키기 위해 실제 문제를 숨기거나 검증을 약화하지 않는다.
- 타입 오류를 피하려고 `any`, `as unknown as`, 불필요한 non-null assertion을 남용하지 않는다.
- 비밀키, 토큰, 개인 정보, 운영 환경 값을 코드나 로그에 노출하지 않는다.

## 목표 기반 실행

작업을 시작할 때 내부적으로 다음을 정리한다.

- 해결해야 할 문제는 무엇인가?
- 성공 조건은 무엇인가?
- 어떤 파일을 수정해야 하는가?
- 어떤 검증을 수행할 수 있는가?
- 변경 후 기존 동작이 유지되어야 하는 부분은 무엇인가?

버그 수정의 경우:

- 가능하면 먼저 실패 조건을 재현한다.
- 실패 원인을 설명할 수 있을 만큼만 조사한다.
- 가장 작은 수정으로 해결한다.
- 동일 문제가 재발하지 않도록 테스트를 추가하거나 기존 테스트를 보강한다.

기능 추가의 경우:

- 기존 패턴을 따른다.
- public API, 데이터 구조, UI 동작의 변경 범위를 명확히 한다.
- 엣지 케이스는 필요한 만큼만 처리한다.
- 요청되지 않은 설정 옵션이나 확장 포인트를 만들지 않는다.

리팩터링의 경우:

- 동작 변경이 없어야 한다.
- 리팩터링 전후의 검증 방법을 명확히 한다.
- 리팩터링과 기능 변경을 한 작업에 섞지 않는다.

## 테스트와 검증

변경 후 가능한 가장 작은 검증부터 수행한다.

우선순위:

1. 수정한 코드와 직접 관련된 단위 테스트
2. 관련 패키지 또는 모듈 테스트
3. 타입체크
4. 린트
5. 빌드
6. 전체 테스트

검증 명령이 실패하면:

- 실패가 이번 변경 때문인지 확인한다.
- 이번 변경으로 발생한 실패라면 수정한다.
- 기존 실패로 보이면 그대로 숨기지 말고, 최종 응답에 명시한다.

검증을 실행하지 못했다면:

- 실행하지 못한 이유를 명확히 말한다.
- 대신 수행한 수동 검증이나 코드 검토 내용을 설명한다.

## 응답 방식

작업 완료 후에는 다음을 간결하게 보고한다.

- 변경한 내용
- 수정한 파일
- 수행한 검증
- 남은 리스크 또는 확인이 필요한 부분

장황한 설명보다 실제 변경 사항과 검증 결과를 우선한다.
